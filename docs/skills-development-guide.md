## 願景與目標

### 從「聰明個體」到「組織能力」

```
┌─────────────────────────────────────────────────────────────────┐
│  Before: 聰明個體              After: 組織能力                  │
│  ─────────────────────────     ─────────────────────────        │
│  每個人 + AI = 各自發揮        團隊 + Agent Skills = 統一標準   │
│       ↓                              ↓                          │
│  • 結果不一致                  • 結果可預期                     │
│  • 經驗不累積                  • 經驗可沉澱                     │
│  • 離職就歸零                  • 人走知識留                     │
└─────────────────────────────────────────────────────────────────┘
```

### 三大核心價值

| 價值       | 對比            | 具體表現                               |
| ---------- | --------------- | -------------------------------------- |
| **可治理** | vs 黑箱、不可控 | 有規範、有邊界、可審計、可追蹤、可稽核 |
| **可擴充** | vs 一次性、孤立 | 可新增、可組合、可升級、可跨專案複用   |
| **可交接** | vs 依賴個人     | 不綁人、可傳承、新人可接手、知識不流失 |

### 解決的核心問題

```
問題                          →  AI 解決方案
────────────────────────────────────────────────────
需求來回溝通耗時              →  Requirement Clarifier
改動影響範圍難評估            →  Impact Analyzer
Code Review 品質不一致        →  PR Review Assistant
安全漏洞容易遺漏              →  Security Review
上線回滾沒有標準流程          →  Migration & Rollback Planner
技術債累積沒人管              →  Tech Debt Inventory
```

### 開發流程覆蓋

```
需求分析 → 架構建議 → 程式碼生成 → 測試生成 → Commit & PR → Review / Security → Release & Deploy
   ↓          ↓           ↓           ↓           ↓              ↓                ↓
 Skills     Skills      Skills      Skills     Skills         Skills           Skills
  1-4        5-8         9-12       13-16       17           17-20            21-29
```

## Agent Skills 的價值定位

### 六層價值模型

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent Skills 價值層                       │
├─────────────────────────────────────────────────────────────┤
│  📋 規範層：行為準則、作業標準、品質門檻                    │
│  📚 知識層：參考知識、經驗沉澱、最佳實踐                    │
│  🔧 工具層：自動化工具、重複使用元件                        │
│  🔗 協作層：團隊橋樑、統一語言、減少摩擦                    │
│  📊 數據層：執行記錄、流程優化、持續改善                    │
│  🛡️ 防護層：風險預警、合規稽核、最後防線                    │
└─────────────────────────────────────────────────────────────┘
```

### 各層價值說明

| 層級         | 說明                               | 對應 Skills 範例                                   |
| ------------ | ---------------------------------- | -------------------------------------------------- |
| **📋 規範層** | 定義「該怎麼做」的標準，確保一致性 | PR Review、Security Review、Style Fixer            |
| **📚 知識層** | 把 Senior 經驗轉化為可執行規則     | Architecture Draft、DB Schema Designer             |
| **🔧 工具層** | 自動化重複性工作，提升效率         | Code Generator、Test Writer、CI/CD Builder         |
| **🔗 協作層** | 減少角色間的理解落差               | Requirement Clarifier、API Contract、Release Notes |
| **📊 數據層** | 收集執行數據，驅動持續改善         | KPI Analyzer、Tech Debt Inventory                  |
| **🛡️ 防護層** | 在問題發生前偵測，守住底線         | Secret Leak Detector、Regression Risk Detector     |

### 組織規模化效益

| 情境         | 效果                          |
| ------------ | ----------------------------- |
| 小團隊做大事 | 3 人團隊 + Skills = 10 人產能 |
| 多專案複用   | 一次開發，N 個專案用          |
| 快速擴編     | 新人 + Skills = 快速達標      |
| 人員流動     | 知識留在 Skills，不隨人走     |

## 開發策略

### 設計原則：對應三大核心價值

#### 🔒 可治理（Governable）

| 原則                      | 說明                            | 實踐方式            |
| ------------------------- | ------------------------------- | ------------------- |
| **先 Review 再 Auto-fix** | 先做分析/審查類，再做自動修改類 | 降低風險、建立信任  |
| **先建議再自動**          | 輸出「建議」而非直接改動        | 讓人保有決定權      |
| **可審計**                | 每次執行都有記錄                | Log、版本、變更追蹤 |
| **有邊界**                | 每個 Skill 只做一件事           | 職責單一、可控可測  |

#### 🔧 可擴充（Extensible）

| 原則             | 說明                              | 實踐方式           |
| ---------------- | --------------------------------- | ------------------ |
| **模組化設計**   | Skill 可獨立運作、可組合          | 鬆耦合、標準介面   |
| **漸進式自動化** | 建議 → 半自動 → 人工確認 → 全自動 | 逐步提升自動化程度 |
| **可配置**       | 同一 Skill 可適應不同專案/團隊    | 參數化、規則可調整 |
| **可升級**       | 新版本可平滑替換舊版本            | 向後相容、版本管理 |

> 可擴充不只代表 Skill 變多，而是：團隊中『需要人腦判斷的情境』持續變少。

#### 🤝 可交接（Transferable）

| 原則         | 說明                       | 實踐方式           |
| ------------ | -------------------------- | ------------------ |
| **知識外顯** | 把隱性經驗轉為顯性規則     | 文件化、規則化     |
| **不綁個人** | Skill 不依賴特定人才能運作 | 標準化輸入輸出     |
| **易於理解** | 新人能快速理解 Skill 用途  | 清楚的命名與說明   |
| **可學習**   | 從 Skill 輸出學習標準做法  | 輸出包含解釋與建議 |

> 可交接的最低標準定義：在沒有原決策者參與的情況下，Skill 仍能提供「不低於團隊最低品質門檻」的判斷與建議。

### Skill 開發三問

每個 Skill 開發前必須回答：

1. **什麼時候用？** — 觸發時機與場景
2. **誰在用？** — 目標使用者（RD / PM / Tech Lead）
3. **輸出給誰看？** — 產出物的消費者

### Skill 類型分級

| 類型                       | 風險  | 開發難度 | 建議順序 | 治理要求 |
| -------------------------- | :---: | :------: | :------: | :------: |
| 🔍 **分析類**（只讀、不改） |  低   |    中    |   先做   |    低    |
| 📝 **建議類**（產生建議）   |  低   |    中    |   次之   |    中    |
| ✏️ **輔助類**（產生草稿）   |  中   |   中高   |   再次   |   中高   |
| 🤖 **自動類**（直接修改）   |  高   |    高    |   最後   |    高    |

決策原則補充：
- 🔍 分析類：Skill 提供風險與事實，人類保有最終決策權
- 📝 建議類：Skill 提供方案與取捨，人類需明確接受或拒絕
- ✏️ 輔助類：Skill 與人共同產出，結果需經人工確認
- 🤖 自動類：僅在「已被授權且可回滾」的範圍內執行

### 觸發設計限制與建議

#### 觸發條件應從「意圖 + 對象」做限制

Skill 的 description 是 Agent 選擇是否啟動的依據。寫得太寬會誤觸發，寫得太窄會叫不到。

| 原則 | 說明 |
|------|------|
| **意圖 + 對象缺一不可** | description 應同時要求使用者有明確意圖（如「review」「分析」）和明確對象（如「PR #47」「staged diff」「這份需求」），兩者缺一不可 |
| **正向匹配為主** | description 負責「什麼情況該想到這個 skill」，複雜排除邏輯放 SKILL.md 內的觸發守門 |
| **能力邊界誠實** | description 宣稱的能力範圍必須和 skill 實際的 workflow / 工具能力一致 |
| **不放孤立動作詞** | 像 `approve`、`request changes`、`inline comment` 這類詞，單獨出現容易誤觸發，應由 SKILL.md 守門邏輯處理 |

#### Skill 數量多時的風險

當安裝的 skill 數量增加，會出現兩個問題：

1. **誤觸發風險上升**：多個 skill 的 description 可能互相重疊，Agent 選錯 skill 的機率增加
2. **Token 成本上升**：Agent 需要在每次對話開始時載入所有 skill 的 metadata（name + description），skill 越多，固定 overhead 越大

#### 應對策略：Skill vs Slash Command

不是所有能力都適合做成 skill（自動觸發）。依使用頻率與觸發精準度，選擇合適的實現方式：

| 面向 | Skill（自動觸發） | Slash Command（手動觸發） |
|------|-------------------|--------------------------|
| 觸發方式 | Agent 依 description 自動判斷 | 使用者手動輸入 `/command` |
| 適合場景 | 高頻、觸發條件明確、不易與其他 skill 混淆 | 低頻、觸發條件模糊、或與其他 skill 容易混淆 |
| 誤觸發風險 | 有（description 重疊時） | 無（使用者主動選擇） |
| Token 成本 | 每次對話都載入 metadata | 僅在呼叫時載入 |
| 使用門檻 | 低（自然語言觸發） | 中（需知道 command 名稱） |

**建議決策流程**：

1. 這個能力的使用頻率高嗎？ → 低頻考慮 slash command
2. 觸發條件是否明確、不易與其他 skill 混淆？ → 容易混淆考慮 slash command
3. 安裝的 skill 總數是否已經偏多（> 10 個）？ → 邊緣的 skill 考慮改為 slash command
4. 使用者是否已經習慣用特定指令呼叫？ → 習慣已養成就維持 slash command

> 一般建議：核心高頻 skill 維持自動觸發；低頻、專用、或容易誤觸發的能力改為 slash command，減少 description 競爭與 token 浪費。

### 1. 設計 (Architecture / Detailed Design)

#### 5) Architecture Draft Writer

| 項目       | 內容                                                   |
