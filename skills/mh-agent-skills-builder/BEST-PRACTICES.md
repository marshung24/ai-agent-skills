# 最佳實踐

## Description 撰寫

### 三段式結構

```yaml
description: [核心功能]。當用戶[高信號使用時機]時使用。[能力邊界/不適用情境]。
```

範例：
```yaml
description: 程式碼審查（code review / 代碼審核 / 程式審核）。當用戶明確要求 review/審核程式碼改動，且審查對象是目前改動、未 commit 的內容、staged/unstaged diff、特定檔案、commit、branch，或已提供 GitHub pull request 編號/URL 時使用。支援自我 review 與 PR review；一般寫法討論、除錯、設計建議不適用。
```

### 適合寫 / 不適合寫

| 適合寫進 description | 不適合寫進 description |
|---------------------|----------------------|
| 核心任務名稱 | 太細的分支規則 |
| 高信號輸入與對象 | 太多動作型近義詞 |
| 適用範圍與邊界 | 補問邏輯 |
| 簡短的不適用說明 | 排除條件的完整清單 |

### 反模式

| 反模式 | 說明 | 後果 |
|--------|------|------|
| 太模糊 | 只有功能名，沒有使用時機 | 不觸發或亂觸發 |
| 太窄 | 只寫單一句型，同義表達無法匹配 | 使用者換個說法就叫不到 |
| 太寬 | 塞太多近義詞、動作詞 | 誤觸發，與守門規則打架 |
| 能力失真 | 寫了尚未支援的場景 | 使用者被誤導 |
| 邏輯打架 | description 與 SKILL.md 守門條件不一致 | 行為矛盾 |

錯誤範例：
- `我可以幫助您處理 PDF` ← 第一人稱
- `處理文件` ← 太模糊
- `PDF 處理功能` ← 沒說明使用時機
- `GitHub/GitLab PR review` ← 實際只支援 GitHub，能力失真

### Description 設計步驟

1. 先定義 skill 的最小成立條件（缺什麼就不該啟動）
2. 列出高信號觸發句（真正希望模型高機率想到 skill 的句型）
3. 列出應排除的相似句（表面像但不該觸發的輸入）
4. 最後才收斂成三段式 description
5. description 寫的能力範圍必須和實際 workflow/工具能力一致

> 不要太早寫 description。先把最小成立條件、高信號觸發、排除句寫出來，再回頭收斂。

## 啟動條件設計

### 何時需要守門

守門主要用於以下 skill：
- 功能詞與日常用語重疊（如「review」「看一下」）
- 有多模式分流（如自我 review / PR review）
- 操作有副作用（寫入外部系統、發送訊息）

功能單一且不會與日常對話混淆的 skill（如 PDF 提取、格式轉換）可以不設守門。

### 三層啟動分類

| 層級 | 條件 | 行為 |
|------|------|------|
| 直接啟動 | 意圖明確 + 對象明確 | 進入 skill |
| 補問後啟動 | 有意圖但缺對象 | 補問，不自動假設 |
| 不啟動 | 泛泛討論 / 不符合 skill 範疇 | 走一般對話 |

### 常見誤觸發案例

| 使用者輸入 | 真正意圖 | 不該觸發的原因 |
|-----------|---------|--------------|
| 「幫我看這段 code 好不好」 | 設計建議 | 不涉及 diff、風險掃描 |
| 「這樣改可以嗎」 | 確認方向 | 不需要完整 review 流程 |
| 「這個 PR 什麼時候 merge 的？」 | 查資訊 | 含 PR 但動詞是查詢 |

### 灰色地帶

「這段有沒有 bug」「檢查一下」這類句子，意圖取決於上下文。處理方式：預設不觸發，使用者如果想要正式執行會進一步說明。

## Token 經濟性

### 核心問題

如果使用者只給一個很小的任務，skill 的固定指引成本是否已經比任務本體更大？回答「是」通常代表結構該重整。

### 起始預算（heuristic）

以下數字來自 mh-code-review 的重構經驗，作為起始參考，非硬上限：

- SKILL.md 超過 100 行時，應檢查是否有內容該下沉到 references
- 盡量控制每次任務的 Read 次數（起始目標 ≤ 3 次，每增加需有明確理由）
- 單一 references 檔 < 50 行時，考慮合併到相鄰檔案

### 零工作量短路

在載入 references 前，先確認有事可做。例：先跑 `git diff`，空 diff 時直接回報，不載入 references。

### 條件載入規則

連結可作為索引導航，但不可把連結本身視為載入控制機制。載入條件要用明確文字寫出：

```markdown
## 載入規則
- 每次都讀 references/framework.md
- 模式 A 時讀 references/mode-a.md，不讀 mode-b
- 模式 B 時讀 references/mode-b.md，不讀 mode-a
```

### 重複內容 = 技術債

同一段邏輯只在一個地方寫。SKILL.md 放一行摘要 + 指標，references 放完整內容。重複會導致修改時漏改一邊（實際案例：SKILL.md 殘留舊用語，references 已更新）。

### A/B 對照

| 維度 | 版本 A（胖版） | 版本 B（精簡版） |
|------|--------------|----------------|
| SKILL.md | 125 行，含完整表格和步驟 | ~80 行，只做路由 |
| Reference | 6 個小檔，每次全讀 | 3 個合併檔，每次讀 2-3 個 |
| 輸出模板 | 獨立拆檔 | 併入各自 workflow |
| 載入規則 | 靠 markdown 連結暗示 | 明確文字指令 |
| 空跑處理 | 載入所有 references 才發現空 | 先跑檢查，空時不載入 |
| 重複內容 | ~40% | 0% |

### Token 節省檢查清單

```
- [ ] SKILL.md 是否超過 100 行？超過時是否有內容該下沉？
- [ ] SKILL.md 與 references 有無重複內容？
- [ ] 每次任務需要幾次 Read？是否可減少？
- [ ] 是否有前置條件可短路（避免空跑）？
- [ ] 載入規則是否用明確文字寫，而非只靠連結？
- [ ] < 50 行的 references 檔是否該合併？
```

## 邊界一致性檢查

交付前 checklist：

```
- [ ] description 的能力宣稱，是否全部有對應 workflow？
- [ ] 是否有動作詞被寫進 description，但缺少必要對象約束？
- [ ] SKILL.md 與 references 是否出現重複規則？
- [ ] 分流規則是否能涵蓋 description 列出的所有場景？
- [ ] 排除場景是否有明確寫出，而不是只靠模型自行理解？
- [ ] README、SKILL.md、references 的術語是否一致？
```

## 命名

推薦動名詞：`processing-pdfs`、`analyzing-data`、`building-reports`

避免：
- 模糊名稱：`helper`、`utils`
- 保留字：`claude-xxx`、`anthropic-xxx`

## 自由度設定

- 多種方法皆可 → 高自由度，文字說明
- 有首選模式 → 中自由度，虛擬碼或帶參數腳本
- 操作脆弱易錯 → 低自由度，精確腳本，禁止修改

## 工作流程

複雜任務提供清單：
```
- [ ] 步驟 1：分析輸入
- [ ] 步驟 2：驗證格式
- [ ] 步驟 3：執行處理
- [ ] 步驟 4：驗證輸出
```

## 反饋循環

關鍵操作加入驗證：
1. 執行操作
2. 立即驗證
3. 失敗 → 修復，返回步驟 1
4. 驗證通過才繼續

## 避免

- Windows 路徑格式（用 `/`）
- 太多選項造成困惑
- 假設套件已安裝
- 時間敏感資訊
- 深層嵌套參考
- 術語不一致

## 交叉驗證（建議）

完成 skill 後，可用不同 AI agent 交叉 review。不同 agent 各有盲點，交叉驗證能補上單一 reviewer 的遺漏。
