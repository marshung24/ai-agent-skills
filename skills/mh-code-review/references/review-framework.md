# 審查框架與檢查清單

## 審查原則：Code-first, Finding-first

Review 的核心是**讀懂改動意圖，再判斷風險**，不是逐條比對規則。

1. 先讀 PR description / Jira / commit message，理解要解決的問題與設計思路
2. 讀 diff，確認改動範圍是否與需求一致（有無混入不相關改動）
3. 從高風險往低風險掃描
4. 找到問題時，標注嚴重度（🔴🟡🟢）並給具體修正建議
5. 沒問題就 LGTM，不要為了寫 review 而硬找問題
6. 風格意見不能偽裝成 blocking issue
7. 每個 finding 都要說清楚「為什麼是問題」
8. 不確定是否為問題時，用「確認：...？」提問而非直接斷言

## 審查優先序

按風險高低依序檢查，前面的比後面的重要：

| 優先級 | 面向 | 要問的問題 |
|--------|------|-----------|
| 1 | **行為回歸** | 這個改動會不會破壞現有功能？有沒有漏改的地方？ |
| 2 | **安全風險** | 有沒有注入、XSS、硬編碼密碼、敏感資訊暴露？ |
| 3 | **邏輯缺陷** | 邊界情況、null handling、off-by-one、例外處理？ |
| 4 | **架構一致性** | 符不符合現有 pattern？有沒有走捷徑繞過框架機制？ |
| 5 | **可維護性** | 下一個人看得懂嗎？命名清楚嗎？複雜度合理嗎？ |
| 6 | **效能** | 有沒有明顯的 N+1、重複計算、不必要的 I/O？ |

## 嚴重度定義

| 標記 | 說明 | 自我 review | PR review |
|------|------|------------|-----------|
| 🔴 | 必須修正：功能錯誤、安全漏洞、資料遺失風險 | 立即修改 | REQUEST_CHANGES |
| 🟡 | 建議修正：效能問題、錯誤處理不足、潛在邊界問題 | 提醒，可含早期建議 | 保守，避免升格為阻擋 |
| 🟢 | 可選優化：命名優化、程式碼風格、文件補充 | 通常省略 | COMMENT |

## Blocking / Non-blocking 判準

**Blocking（🔴，必須修正才能合併）**：
- 改動會導致現有功能行為改變或中斷
- 存在安全漏洞（注入、XSS、敏感資訊暴露）
- 明確的邏輯錯誤（條件反轉、off-by-one、未處理 null）
- 高機率造成維運事故的問題

**Non-blocking（🟡🟢，建議但不阻擋合併）**：
- 效能可改善但不影響正確性
- 命名或結構可更清楚但不影響理解
- 缺少測試但功能本身正確
- 風格偏好差異

## 兩種模式的門檻差異

| 面向 | 自我 review | PR review |
|------|------------|-----------|
| 🔴 門檻 | 相同 | 相同 |
| 🟡 門檻 | 寬鬆，可含早期提醒與設計建議 | 保守，避免把偏好升格為阻擋 |
| 🟢 處理 | 通常省略 | 可列為 COMMENT |
| REQUEST_CHANGES | 不適用 | 僅限 🔴 + 少數嚴重 🟡 |

## Review 禁忌

- 為了寫 review 而硬找問題
- 把風格偏好包裝成 blocking issue
- 空泛批評不附修正建議
- 對人不對事
- 未讀 diff 就下結論
- 擴張到本次 PR 範圍外的無關模組或風格差異

## 檢查清單

依審查優先序排列。標有 **[PR]** 的項目僅適用於 PR review，自我 review 時可跳過。

### 🔴 高風險（必須確認）

**行為回歸**
- [ ] 改動是否影響現有功能的行為
- [ ] 相關聯的程式碼是否同步調整
- [ ] 是否有漏改的 caller / consumer
- [ ] 回傳值或資料結構變更是否影響下游

**安全風險**
- [ ] 輸入是否驗證與過濾
- [ ] 是否有 SQL injection / XSS / command injection 風險
- [ ] 是否有硬編碼密碼、API key、敏感資訊暴露
- [ ] 權限檢查是否正確（越權存取）
- [ ] 檔案上傳是否有類型/大小限制

**邏輯缺陷**
- [ ] 邊界情況是否處理（null、空陣列、零值、超長輸入）
- [ ] 錯誤處理是否完善（例外、失敗路徑、重試）
- [ ] 條件判斷是否正確（off-by-one、邏輯運算子）
- [ ] 型別轉換是否安全（隱式轉換、精度遺失）

### 🟡 中風險（建議確認）

**PR 範圍 [PR]**
- [ ] 改動範圍是否與 PR description / Jira 一致
- [ ] 是否混入不相關改動（格式化、重構混入 feature PR），建議作者拆分
- [ ] 分支是否已 rebase 到目標分支最新節點？落後時要求開發者先 rebase

**架構一致性**
- [ ] 是否符合現有 codebase 的 pattern 與慣例
- [ ] 是否有繞過框架機制的捷徑寫法
- [ ] 新增的依賴是否合理且必要
- [ ] 資料流是否符合分層架構

**可維護性**
- [ ] 命名是否清晰、意圖是否明確
- [ ] 複雜邏輯是否有拆分或註解
- [ ] 是否有重複代碼可抽取
- [ ] 函數長度與職責是否合理

**部署風險**（涉及 DB migration / API 契約 / 跨 repo / 環境變數時檢查）
- [ ] 是否有 DB migration？是否可安全 rollback？
- [ ] 前後端或跨 repo 是否需同步部署？有無時間差風險？ [PR]
- [ ] 此改動是否可安全 revert（無不可逆操作如 DROP COLUMN）？ [PR]
- [ ] 是否新增或修改環境變數（.env*）？對應的 .env.example 是否同步更新？

### 🟢 低風險（可選）

**效能**
- [ ] 是否有明顯的 N+1 查詢或重複計算
- [ ] 是否有不必要的 I/O 或大量資料載入
- [ ] 迴圈中是否有可提前中斷的條件
- [ ] 快取策略是否合理

## 常用 Review 術語

| 術語 | 意義 |
|------|------|
| LGTM | Looks Good To Me — 核准，沒問題 |
| NIT | Nitpick — 小問題，不阻擋合併 |
| PTAL | Please Take A Look — 請查看（通常用於請求 re-review） |
| WIP | Work In Progress — 尚未完成，先不 review |
| RFC | Request For Comments — 徵求意見，歡迎討論 |
